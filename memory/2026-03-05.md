## 🎯 关键质疑（必须记住）

### 质疑1: quality-agent 的实现方式

**质疑**: 为什么用脚本调用 API，而不是使用 AI 模型？

**纠正**: quality-agent 应该使用自己的 AI 模型分析内容，而不是脚本调用 API。

**原则**: AI 模型调用是 Agent 的核心能力，不应该外包给 Skill。

---

### 质疑2: 可复用的脚本应该封装为 Skill

**质疑**: 为什么在多个地方写相同的脚本逻辑？

**纠正**: 可复用的逻辑应该封装为 Skill，统一维护。

**原则**: 单一功能的脚本 → 封装为 Skill（在≥2个地方使用）

---

### 质疑3: 预先定义的 orchestrate 逻辑

**质疑**: 预先定义的 orchestrate 逻辑不违背智能协同的初衷吗？

**纠正**: 预先定义固定流程不是智能协同，只是"脚本式 Agent 调用"。

**原则**: Main Agent 应该使用 AI 模型智能决策，动态调用 Agents，而不是预先定义固定流程。

---

## 🔍 深层理解

### 错误的设计

```python
# ❌ 预先定义的固定流程
def orchestrate(user_input):
    step1 = sessions_spawn(requirement-agent)
    step2 = sessions_spawn(research-agent)
    step3 = sessions_spawn(content-agent)
    ...
```

**问题**:
- 缺乏智能判断
- 总是按相同顺序
- 不是真正的协同

### 正确的设计

```python
# ✅ Main Agent 使用 AI 模型智能决策
def handle_user_request(user_input):
    # 1. 理解需求（AI 模型）
    # 2. 智能决策需要哪些 Agents（AI 模型）
    # 3. 动态调用 Agents
    # 4. 根据结果调整策略（AI 模型）
    # 5. 整合结果（AI 模型）
```

**优势**:
- 智能决策
- 灵活调整
- 真正的协同

## 🎯 核心教训

### 1. Agent ≠ 脚本

- Agent = 智能（AI 模型 + 决策）
- 脚本 = 工具（固定逻辑）

### 2. 智能协同 ≠ 固定流程

- 智能协同 = AI 模型 + 动态决策
- 固定流程 = 脚本式调用

### 3. Main Agent 的本质

- Main Agent 是一个智能体
- 应该使用 AI 模型智能决策
- 不是预先定义的流程执行器

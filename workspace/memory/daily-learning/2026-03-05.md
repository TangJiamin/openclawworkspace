## 学习内容

### 1. 第一性原理的核心思想

**定义**:
> 剥离所有表象和假设，回到最基础的事实，从零开始重新构建解决方案

**三个关键步骤**:
1. **理解本质** - 问题背后真正的需求是什么？
2. **拆解到基础** - 能不能再分解？最基础的事实是什么？
3. **重新构建** - 从基础事实出发，设计最优解

**核心心态**:
- ❌ 不要因为"以前这样做"就继续这样做
- ❌ 不要接受"一直以来都是这样"
- ✅ 多问几次"为什么"
- ✅ 质疑所有假设

---

### 2. 第一性原理的工作方式

**收到任务时的 5 步流程**:

1. **暂停** 🛑
   - 不要立即动手
   - 不要套用模板
   - 不要参考"上次怎么做的"

2. **提问** 🤔
   - 问题的本质是什么？
   - 用户的真正需求是什么？
   - 有没有更好的解决方案？

3. **拆解** 🔨
   - 第一性原理：最基础的事实是什么？
   - 还能再分解吗？
   - 各部分之间的依赖关系是什么？

4. **构建** 🏗️
   - 从基础事实出发
   - 重新设计解决方案
   - 对比多个方案，选择最优

5. **验证** ✅
   - 这个方案真的解决了本质问题吗？
   - 是最优解吗？
   - 有没有更好的方式？

---

### 3. 寻找最优解的判断标准

**可行 ≠ 最优**

判断标准：
- **效率** (时间/资源)
- **灵活性** (适应性)
- **可维护性** (清晰度)
- **可扩展性** (未来需求)

**例子**:
```
可行方案: 创建一个 visual-agent 处理所有视觉内容
最优方案: 分离 visual-agent 和 video-agent，支持并行执行

判断依据:
- 效率: 并行执行更快 ✅
- 灵活性: 独立优化更容易 ✅
- 可维护性: 职责分离更清晰 ✅
- 可扩展性: 可以单独扩展某一功能 ✅
```

---

### 4. 明确假设，验证假设

**列出所有假设，逐一验证**:

```
假设1: 用户需要并行执行 → 验证: 是的，明确要求
假设2: 图片和视频可以独立生成 → 验证: 是的，无依赖
假设3: 分离后可以并行 → 验证: 是的，Promise.all
```

**关键**: 如果假设被证明是错的，立即调整方案

    → 使用 host.docker.internal:7897
    → 验证成功后再持久化配置

Step 5: 验证 ✅
  - 是否解决了本质问题？
    → 是的，容器可以访问宿主机代理
  - 是最优解吗？

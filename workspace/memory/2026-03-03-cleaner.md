## 实施方案

### 第一性原理分析

**本质问题**: 容器文件随时间积累，需要定期清理

**最优解**: Cron Job + 专用清理 Agent + 清理 Skill

### 架构设计

```
Cron Job (每天凌晨 3:00)
  ↓
cleaner-agent (协调者)
  ↓
cleanup Skill (工具)
  ├─ 扫描文件系统
  ├─ 生成清理预览
  ├─ 执行清理操作
  └─ 生成清理报告
  ↓
飞书通知
```

## 技术亮点

### 1. 第一性原理设计

不从"怎么做清理"出发，而是从"为什么要清理"出发：
- 问题本质：文件积累
- 真正需求：自动化 + 安全性
- 最优解：分层架构（Agent + Skill）

### 2. 分层架构

- **Agent（协调者）**: 负责定时触发和通知
- **Skill（工具）**: 负责具体的清理逻辑
- **Cron Job（调度）**: 负责定时执行

### 3. 白名单保护

三层保护机制：
1. 文件名匹配
2. 路径匹配
3. 特殊规则

### 4. 干跑模式

首次运行支持干跑模式，预览清理结果后再执行。

### 5. 详细报告

每次清理生成 Markdown 报告，包含：
- 扫描结果
- 删除文件列表
- 保留文件说明
- 统计信息

## 关键决策

### 1. 为什么不直接写一个脚本？

**可行方案**: 单一脚本完成所有清理
**最优方案**: Agent + Skill 分层

**原因**:
- ✅ 可复用：cleanup Skill 可被其他 Agent 调用
- ✅ 可扩展：未来可添加更多清理规则
- ✅ 可测试：独立测试每个组件
- ✅ 可维护：逻辑分离，易于维护

### 2. 为什么使用 Cron Job 而不是系统 crontab？

**可行方案**: 系统级 crontab
**最优方案**: OpenClaw Cron Job

**原因**:
- ✅ 原生集成：与 OpenClaw 系统无缝集成
- ✅ 飞书通知：直接支持飞书通知
- ✅ Agent Turn：支持在独立 session 中执行
- ✅ 统一管理：与其他 OpenClaw 任务统一管理

### 3. 为什么将 extensions 加入白名单？

**原因**:
1. 用户明确要求
2. 大小可控（1.3GB qqbot node_modules 除外）
3. 可能未来会使用
4. 避免误删导致功能缺失

**未来优化**:
- 如果确认不使用某个 extension，可以手动删除
- 添加 extension 使用率统计
- 自动清理长期未使用的 extension

### 4. 为什么只保留 4 个核心文档？

## 遇到的问题

### 问题 1: Bash 脚本错误处理

**问题**: `set -euo pipefail` 导致 find 命令失败时退出

**解决**:
1. 改用 `set -eo pipefail`（去掉 `-u`）
2. 在 find 命令后添加 `|| true`
3. 最终采用简化版脚本，避免复杂错误处理

### 问题 2: 文件计数错误

**问题**: `while` 循环中的变量计数不正确

**解决**: 简化脚本，直接输出文件列表，不计数

### 问题 3: 飞书通知集成

**问题**: Agent 中无法直接调用 message 工具

**解决**: 
1. 使用 Cron Job 的 `delivery.announce` 自动通知
2. 或者输出到 stdout，让主 Agent 处理
